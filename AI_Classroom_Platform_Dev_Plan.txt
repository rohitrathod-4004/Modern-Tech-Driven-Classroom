Section 1 — System Redesign & Architecture
1.1 Overall Architecture Philosophy
The upgraded architecture follows a Modular Monolith-First approach: strong internal module boundaries with well-defined interfaces, but deployed as a small number of services (not dozens of microservices). This gives you the speed of a monolith with the upgrade path of microservices.
Architecture Principles
  Modular monolith on the backend — clean module boundaries without over-segmentation
  Domain-driven folder structure — auth, courses, lectures, transcripts, ai, search
  Service-layer pattern — business logic lives in services, not route handlers
  Repository pattern — data access abstracted behind repositories
  Event-driven processing — async AI pipelines decoupled from request/response cycle
  Contract-first API design — TypeScript interfaces shared between frontend and backend
1.2 High-Level System Architecture
The system is composed of four primary runtime processes:

1.3 Data Flow — Complete Request Lifecycle
Recording & Transcription Flow
1. Student starts recording — MediaRecorder captures mic at 3-second chunks.
2. Each chunk is compressed and persisted to IndexedDB immediately (offline-first).
3. Upload queue picks up chunks and POST to /api/lectures/:id/chunks with JWT auth.
4. Node.js validates auth, associates chunk with lecture+user, saves metadata to MongoDB.
5. Node.js streams WAV to Python AI service via internal HTTP.
6. Whisper transcribes and returns text + timestamps + language.
7. Node.js stores TranscriptChunk document in MongoDB with position/timestamp metadata.
8. Frontend polls GET /api/lectures/:id/transcript and renders incrementally.
9. After lecture ends, backend triggers async AI pipeline: summarization + embedding generation.
10. AI service generates embeddings using sentence-transformers and upserts to Qdrant.

The frontend follows a feature-sliced architecture with clear separation between infrastructure code (API clients, stores) and feature code (recording, dashboard, classroom, assistant).

The Python FastAPI service expands from pure Whisper transcription to a multi-capability AI service. Each capability is a separate router/module, allowing independent scaling if needed.

1.7 RAG Architecture Overview
The Retrieval-Augmented Generation system is the heart of the AI assistant. It enables students to ask natural language questions and receive answers grounded in actual lecture content.
Ingestion Pipeline: After each lecture session ends, transcript chunks are split into semantic units (~300 tokens with 50-token overlap).
Embedding: Each chunk is embedded using sentence-transformers/all-MiniLM-L6-v2 or text-embedding-3-small (OpenAI).
Storage: Embeddings + metadata (lecture_id, course_id, timestamp, chunk_text) are stored in Qdrant.
Retrieval: On assistant query, the question is embedded and top-K most similar chunks are retrieved.
Augmentation: Retrieved chunks are injected as context into an LLM prompt.
Generation: LLM generates a grounded, cited response. References to lecture timestamps are included.

Section 3 — Authentication System Design
3.1 JWT Architecture Overview
Authentication uses a dual-token strategy: a short-lived access token (15 minutes) and a long-lived refresh token (7 days). This balances security (limited access token window) with UX (no re-login every 15 minutes).
Token
TTL / Storage / Purpose
Access Token (JWT)
TTL: 15 min | Storage: Memory (JS var) | Used on every API call
Refresh Token
TTL: 7 days | Storage: HttpOnly Secure Cookie | Used to mint new access tokens
Why NOT localStorage for tokens?
  localStorage is accessible by any JS on the page — XSS attacks can steal tokens.
  Refresh tokens in HttpOnly cookies cannot be read by JavaScript at all.
  Access tokens in memory are lost on tab close — acceptable since refresh token handles renewal.
3.2 Registration & Login Flow
POST /api/auth/register
Body: { name, email, password, role: 'student'|'teacher' }
Flow:
  1. Validate input (Zod schema)
  2. Check email uniqueness
  3. Hash password (bcrypt, 12 rounds)
  4. Create User document
  5. Generate accessToken + refreshToken
  6. Set refreshToken as HttpOnly cookie
  7. Return accessToken + user profile in body
POST /api/auth/login
Flow:
  1. Find user by email
  2. Compare password (bcrypt.compare)
  3. Generate new token pair
  4. Set cookie + return access token
3.3 Token Refresh Flow
POST /api/auth/refresh
Flow:
  1. Read refreshToken from HttpOnly cookie
  2. Verify JWT signature + expiry
  3. Load user from DB (check isActive)
  4. Generate new accessToken (optionally rotate refreshToken)
  5. Return new accessToken in body
Frontend Interceptor (Axios):
  - On 401 response, auto-call /auth/refresh
  - Retry original request with new token
  - If refresh also fails, redirect to /login

3.5 Auth Middleware
// authenticate.ts — verifies JWT on every protected route
export const authenticate = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token' });
  const payload = jwt.verify(token, process.env.JWT_SECRET);
  req.user = await UserRepository.findById(payload.sub);
  next();
};
// authorize.ts — checks role
export const authorize = (...roles: string[]) => (req, res, next) => {
  if (!roles.includes(req.user.role)) {
    return res.status(403).json({ error: 'Forbidden' });
  }
  next();
};
// Usage:
router.post('/courses', authenticate, authorize('teacher', 'admin'), createCourse);
router.get('/lectures/:id', authenticate, authorize('student','teacher','admin'), getLecture);

Section 4 — AI Architecture & Pipelines
4.1 Post-Lecture AI Pipeline
When a lecture recording ends (teacher clicks Stop), the backend enqueues an async job. This decouples the heavy AI work from the user-facing request cycle. The pipeline runs in the background and updates the lecture's aiProcessingStatus flags as each stage completes.
Post-Lecture AI Pipeline Stages
  Stage 1 — Transcript Assembly: Fetch all TranscriptChunks for lecture, join into full text
  Stage 2 — Semantic Chunking: Re-split into ~300-token chunks with 50-token overlap for RAG
  Stage 3 — Embedding Generation: Batch embed all chunks via sentence-transformers
  Stage 4 — Qdrant Upsert: Store vectors + metadata in Qdrant collection
  Stage 5 — Summarization: Send full transcript to LLM, extract summary/keyPoints/concepts
  Stage 6 — Quiz Generation: Send transcript sections to LLM, generate MCQs + short answers
  Stage 7 — Flashcard Generation: Generate concept-answer cards from key concepts
  Stage 8 — Status Update: Mark lecture aiProcessingStatus fields as complete in MongoDB
4.2 Chunking Strategy
Chunking strategy is critical for RAG quality. Too small = no context. Too large = retrieval noise.
Chunk Type
Configuration
Raw Transcript Chunks
3-second audio chunks from Whisper (preserved as-is in MongoDB)
Semantic RAG Chunks
~300 tokens per chunk, 50-token overlap, sentence-boundary aware
Summary Input
Full transcript concatenated (for lecture-level summaries)
Quiz Input
Topic segments (~1000 tokens each, for topic-specific question generation)

async def generate_embeddings(req: EmbedRequest):
    model = SentenceTransformer('all-MiniLM-L6-v2')  # 384-dim, fast, good quality

4.5 RAG Assistant Pipeline
The AI assistant pipeline implements Retrieval-Augmented Generation. Every student query goes through retrieval before generation — ensuring answers are grounded in actual lecture content.
# chat.py — RAG pipeline
@router.post('/chat')
async def rag_chat(req: ChatRequest):
    # 1. Embed the user's question
    q_embedding = embed_model.encode([req.question])[0]
    # 2. Retrieve top-K similar chunks from Qdrant
    results = qdrant.search(
        collection_name='lecture_chunks',
        query_vector=q_embedding,
        query_filter=Filter(must=[
            FieldCondition(key='course_id', match=MatchValue(value=req.course_id))
        ]),
        limit=5
    )
    # 3. Build context from retrieved chunks
    context_parts = []
    for r in results:
        p = r.payload
        context_parts.append(
            f'[Lecture {p[lecture_id]}, ~{int(p[start_time])}s]: {p[text]}'
        )
    context = '\n\n'.join(context_parts)
    # 4. Build prompt with context
    messages = [
        { 'role': 'system', 'content': RAG_SYSTEM_PROMPT },
        *req.history[-6:],       # last 3 turns for continuity
        { 'role': 'user', 'content': f'Context:\n{context}\n\nQuestion: {req.question}' }
    ]
    # 5. Generate response
    response = await llm_client.chat.completions.create(
        model='gpt-4o-mini', messages=messages, max_tokens=800
    )
    return { 'answer': response.choices[0].message.content, 'sources': results }

Section 5 — Smart Timeline System
5.1 Architecture Overview
The smart timeline synchronizes the transcript with time positions, enabling students to navigate, search, and jump to specific moments. It is built entirely from the timestamp data already captured by Whisper.
5.2 Timestamp Data Model
Each TranscriptChunk already stores startTime and endTime (seconds from lecture start). The timeline is a derived view of these chunks.
// TranscriptChunk (already in DB)
{ chunkIndex, text, startTime, endTime, lectureId }
// Timeline entry (computed, sent to frontend)
{
  chunkIndex: number,
  text: string,
  startTime: number,  // seconds
  endTime: number,
  isHighlighted: boolean,  // matches search query
  topicLabel: string | null,  // AI-detected topic (optional)
}
5.3 Frontend Timeline Implementation
The timeline component is a scrollable list of transcript chunks with time labels. It uses a virtualized list (react-virtual) for performance with long lectures. The audio player position drives the active chunk highlight via a useEffect that finds the chunk whose time range contains the current playback position.
Timeline Features
  Click-to-seek: Clicking a chunk calls onSeek(chunk.startTime) which updates audio player
  Auto-scroll: Active chunk (based on audio position) scrolls into view automatically
  Inline search: Filter bar highlights matching chunks and scrolls to first match
  Topic badges: AI-detected topic labels shown as colored badges on chunk groups
  Copy button: Click chunk to copy text to clipboard
  Share timestamp: Generate sharable link with ?t=seconds parameter
5.4 Topic Segmentation
For lectures that have been AI-processed, the summarization pipeline also identifies topic boundaries. These are stored in the summary document and overlaid on the timeline as visual dividers.

const apiClient = axios.create({ baseURL: import.meta.env.VITE_API_URL });

  res => res,
  async err => {
    if (err.response?.status === 401 && !err.config._retry) {
      err.config._retry = true;

      return apiClient(err.config);
    }
    return Promise.reject(err);

9.3 GPU Deployment Strategy
For the Whisper AI service, GPU access dramatically reduces transcription latency. For v1, the simplest approach is a single GPU VM (e.g., AWS EC2 g4dn.xlarge or Google Cloud n1-standard-4 with T4 GPU). If GPU is unavailable, faster-whisper's CPU mode is still practical for medium models.

9.4 Storage Strategy
In v1, audio chunks are NOT stored long-term. Only transcripts are persisted. This keeps storage costs near-zero. If you need audio replay, use temporary chunk storage (delete after successful transcription) with a 24h TTL cleanup job.

Section 11 — Risks & Challenges
11.1 Risk Register
Risk
Cause
Mitigation Strategy
Whisper accuracy on accents/noise
faster-whisper is strong but not perfect on noisy audio or heavy accents
Use large-v2 model where GPU available; add confidence score threshold; allow manual transcript editing in UI
FFmpeg disk I/O bottleneck
Heavy writes/reads for every 3-second chunk
Use /tmp ramdisk for FFmpeg temp files; pipeline directly from buffer; consider in-memory WebM-to-WAV conversion
RAG quality degradation
Poor chunking or embeddings lead to irrelevant context retrieval
Experiment with chunk size; use chunk overlap; consider hybrid search (keyword + vector); evaluate retrieval quality with test set
LLM API cost overrun
Unexpected usage spikes or long transcripts increase API costs
Set per-user daily limits; cache summaries/quiz (never regenerate if exists); monitor token usage; use GPT-4o-mini not GPT-4o
Cold start latency on AI service
Whisper model loading + sentence-transformer loading on first request can be 30-60s
Pre-load all models at startup; use Docker healthcheck to wait for readiness; consider keeping warm with ping job
MongoDB query performance
Large transcript_chunks collections with no compound indexes
Add compound indexes (lectureId+chunkIndex) on day one; paginate all transcript queries; add DB query monitoring
Multilingual RAG challenges
Embeddings from multilingual transcripts may have lower semantic alignment
Use multilingual-e5-large-instruct embedding model; keep language metadata in Qdrant filter; query in same language
Qdrant data loss
Container restart wipes vectors if volume not configured correctly
Always mount named Docker volume for Qdrant; add Qdrant snapshot job; test disaster recovery before launch
Session security gaps
Leaked refresh tokens allow persistent unauthorized access
Short refresh token TTL (7 days); token rotation on every refresh; server-side refresh token denylist in Redis
Concurrent recording conflicts
Two devices recording same lecture creates duplicate chunks
Enforce single active recording per lecture server-side; use lectureId lock in Redis during recording
Section 12 — Final Recommendations
12.1 Best Architecture Choices
Decision
Recommendation & Rationale
Backend framework
Keep Node.js/Express. It's working. Don't migrate.
State management
Zustand over Redux. Less boilerplate, same power for this scale.
Job queue
BullMQ over raw setTimeout. Persistent, retriable, monitorable.
Embedding model
Start with all-MiniLM-L6-v2 (free, fast). Upgrade to OpenAI ada if quality insufficient.
LLM
GPT-4o-mini for all AI content. Best cost/quality for structured outputs.
Vector DB
Qdrant over Pinecone. Self-hosted, no per-vector billing, good Python client.
Auth storage
HttpOnly cookie for refresh token. Non-negotiable for security.
Async pipeline
Decouple AI processing from lecture end. Never make users wait for LLM calls.
12.2 Priority Order
If you had to ship something usable in 3 weeks, this is the order:
Auth + Course + Lecture model (users need identity before anything else)
Basic dashboard (lecture history is the #1 daily-use feature)
Smart Timeline (this is the core product differentiation vs just transcription)
Summaries (highest perceived value for students, simplest to implement)
Quiz generation (second highest perceived value)
Semantic search (needed to make the knowledge base useful)
RAG assistant (the 'wow' feature — save for last as it's the most complex)
12.3 What NOT to Overengineer
Avoid These Traps in V1
  Do NOT build a microservices architecture. One Node.js process + one Python process is correct.
  Do NOT add GraphQL. REST is simpler, debugging is easier, clients are well-understood.
  Do NOT build a plugin system or abstract everything. YAGNI applies — build what you need.
  Do NOT add a message broker (Kafka/RabbitMQ). Bull+Redis is sufficient for this workload.
  Do NOT pre-optimize MongoDB queries. Add indexes when you have data to profile against.
  Do NOT build a custom embedding model. Use sentence-transformers; it's production-quality.
  Do NOT add caching layers before you measure what needs caching.
12.4 MVP vs Future Scalability Balance
The architecture described in this plan is deliberately designed at the intersection of 'shippable in weeks' and 'upgradeable without rewrite.' The key architectural choices that preserve future scalability:
Service layer pattern in Node.js: you can extract any service into a microservice later without touching routes or models.
Qdrant from day one: adding more vectors is free — no architectural change needed to scale semantic search.
Bull queue: you can add more workers for AI processing without changing the API layer.
Feature-sliced frontend: adding a new feature is isolated; it doesn't require touching other features.
Shared TypeScript types: contract changes are caught at compile time, not at runtime.
12.5 Startup-Grade Implementation Advice
Principal Engineer Advice
  Ship working software first. A complete, stable v1 is worth more than a partial v2.
  Write integration tests for your AI pipeline. LLM outputs are non-deterministic — test the plumbing.
  Log everything in the AI pipeline. When RAG answers are wrong, you need traceability.
  Version your API from day one (/api/v1/...). You will regret it if you don't.
  Guard your LLM API key religiously. One leaked key + one public demo = very large bill.
  Build a simple admin page to view/debug AI pipeline jobs early. It saves hours of debugging.
  Get real users (even 5 teachers) on the system before Phase 3. Their feedback will change priorities.
  Keep your Whisper model size small (base or medium) during development. Use large-v2 in production.
